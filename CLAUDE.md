# MAFIS — Claude Instructions

Multi-Agent Fault Injection Simulator — Bevy 0.18 → WASM. A **fault resilience observatory** for lifelong MAPF: measures how multi-agent systems degrade, recover, and adapt under real-world fault conditions in real-time 3D.

**This is NOT a solver benchmark.** Solver benchmarking is MAPF Tracker's domain. MAFIS measures what happens when things go wrong.

---

## Testing Workflow (ALWAYS follow this order)

**Never go straight to a WASM build to verify a change. Run the fast loops first.**

```bash
# Step 1 — type & borrow check (~5s). Catches 80% of errors instantly.
cargo check

# Step 2 — logic correctness (~7s). 188 tests across core, solver, analysis.
cargo test

# Step 3 — only if Steps 1-2 pass AND the change touches rendering, bridge, or ECS systems.
sh topologies/build-manifest.sh
cargo build --release --target wasm32-unknown-unknown
wasm-bindgen --out-dir web --target web target/wasm32-unknown-unknown/release/mafis.wasm
basic-http-server web
```

**What each step covers:**

| Step | Time | Covers |
|------|------|--------|
| `cargo check` | ~5s | Types, borrow checker, imports |
| `cargo test` | ~7s | Grid, actions, heuristics, PIBT, task scheduling, cascade, ADG, fault metrics, heatmap |
| WASM build | ~2-3 min | Rendering, Bevy ECS systems, JS bridge, visual correctness |

**When WASM build is NOT needed:** pure logic changes (solver tweaks, analysis math, constants). `cargo test` is sufficient.

**When WASM build IS needed:** anything in `render/`, `ui/bridge.rs`, Bevy system ordering, new ECS components.

## Build Pipeline (CRITICAL)

**Always use the binary target**, not the cdylib. The cdylib compiles but the Bevy event loop never starts (0 rAF calls).

```bash
# 0. Rebuild topology manifest (picks up added/removed .json files + syncs to web/)
sh topologies/build-manifest.sh

# 1. Compile
cargo build --release --target wasm32-unknown-unknown

# 2. Bind (hyphens = binary target)
wasm-bindgen --out-dir web --target web target/wasm32-unknown-unknown/release/mafis.wasm

# 3. Optimize (requires binaryen: brew install binaryen)
wasm-opt -Oz --enable-bulk-memory --enable-mutable-globals -o web/mafis_bg.wasm.opt web/mafis_bg.wasm
mv web/mafis_bg.wasm.opt web/mafis_bg.wasm

# 4. Serve
basic-http-server web   # port 4000
```

JS/HTML imports use hyphens: `import init from './mafis.js'`

`Cargo.toml` has both `[lib] crate-type = ["cdylib", "rlib"]` and `[[bin]]`. The binary (`mafis`) is what gets bound — it contains both `fn main()` and all `#[wasm_bindgen]` exports from `lib.rs`.

## Bevy↔JS Bridge (`src/ui/bridge.rs`)

```rust
thread_local! { static BRIDGE: RefCell<BridgeInner> }
```

- `#[wasm_bindgen] pub fn get_simulation_state() -> String` — reads `BRIDGE.outgoing`
- `#[wasm_bindgen] pub fn send_command(cmd: &str)` — writes to `BRIDGE.incoming`
- `sync_state_to_js` (Update) — serializes ECS → JSON → BRIDGE; adaptive interval (90/150/250/500 ms by agent count)
- `process_js_commands` (Update) — drains BRIDGE.incoming → applies to ECS
- JS polls at 100 ms via `setInterval`
- Above `AGGREGATE_THRESHOLD` agents: sends `AgentSummary` instead of per-agent snapshots

## Solver (`src/solver/`)

Only lifelong-capable solvers belong. One-shot solvers (CBS, LaCAM, PBS, LNS2) are archived on `archive/one-shot-solvers` branch.

### LifelongSolver trait (`lifelong.rs`)

All solvers implement `LifelongSolver`. Called every tick via `step()` — solver decides internally whether to replan.

```rust
trait LifelongSolver: Send + Sync + 'static {
    fn name(&self) -> &'static str;
    fn info(&self) -> SolverInfo;
    fn reset(&mut self);
    fn step(&mut self, ctx, agents, cache, rng) -> StepResult; // Replan or Continue
}
```

- `ActiveSolver` resource: `Box<dyn LifelongSolver>`
- Factory: `lifelong_solver_from_name("pibt"|"rhcr_pbs"|"rhcr_pibt"|"rhcr_priority_astar"|"token_passing", grid_area, num_agents)`
- Pre-allocated reusable buffer (`Vec<AgentPlan>`) — zero allocation on replan
- `StepResult::Continue` = zero-cost (counter check only)

### Solvers

| Solver | File | Characteristic |
|--------|------|----------------|
| PIBT | `pibt.rs` | Reactive, one-step priority inheritance; replans every tick |
| RHCR (PBS) | `rhcr.rs` + `pbs_planner.rs` | Windowed PBS with node limit; replans every W ticks |
| RHCR (PIBT-Window) | `rhcr.rs` + `pibt_window_planner.rs` | Unrolled PIBT for H steps; fast, cooperative |
| RHCR (Priority A*) | `rhcr.rs` + `priority_astar_planner.rs` | Sequential spacetime A*; good for moderate density |
| Token Passing | `token_passing.rs` | Decentralized sequential planning via shared TOKEN; ≤100 agents |

### RHCR Architecture (Strategy Pattern)

- `WindowedPlanner` inner trait: PBS, PIBT-Window, Priority A* each implement `plan_window()`
- `RhcrSolver` owns `Box<dyn WindowedPlanner>` + handles windowing + fallback
- `PibtCore` (`pibt_core.rs`): shared algorithm used by standalone PIBT, PIBT-Window, and RHCR fallback
- `RhcrConfig::auto(mode, grid_area, num_agents)`: smart defaults for H, W, node_limit
- Three fallback modes: PerAgent, Full, Tiered (default per RHCR mode, user-overridable)
- Congestion detection: dynamic replan interval shortens when >50% agents stuck
- `WindowAgent.goal_sequence`: multi-goal sequences for windowed planning
- Legacy `MAPFSolver` trait kept for `PibtSolver` (initial loading solve)

### Token Passing Architecture

- `TokenPassingSolver` (`token_passing.rs`): decentralized sequential planning
- TOKEN stores all agents' planned paths as `Vec<IVec2>` position sequences
- Idle agents plan via spacetime A* against ConstraintIndex built from all other TOKEN paths
- Sequential planning order: tasked agents first (PIBT_MAPD-style prioritization)
- Handles fault-induced path divergence: resets token when actual position ≠ expected

## Topology (`src/core/topology.rs`, `topologies/*.json`)

Topologies are defined exclusively as JSON files in `topologies/`. No Rust-generated maps.

- `TopologyRegistry`: loads JSON files from `topologies/` via `manifest.json`
- `ActiveTopology::from_name(id)`: looks up topology from registry, falls back to first available
- `ActiveTopology::from_entry(entry)`: builds topology from a `TopologyEntry`
- `TopologyRegistry::parse_json_value()`: parses JSON → (GridMap, ZoneMap) including `queue_direction`
- `Topology` trait: `fn name()` + `fn generate(seed) -> TopologyOutput`
- `TopologyOutput`: grid + ZoneMap + suggested_agents
- `ZoneMap` resource: pickup_cells, delivery_cells, corridor_cells, queue_lines, zone_type HashMap
- Bridge command: `set_topology "warehouse_medium"|"warehouse_medium"` (any id from registry)
- To add a new topology: create JSON in `topologies/`, run `sh topologies/build-manifest.sh`

## Task Scheduling (`src/core/task.rs`)

| Scheduler | Characteristic |
|-----------|----------------|
| Random | Random pickup/delivery from zone cells |
| Closest | Nearest pickup/delivery by Manhattan distance |

- `TaskLeg` enum (8 states): `Free` → `TravelEmpty` → `Loading` → `TravelToQueue` → `Queuing` → `TravelLoaded` → `Unloading` → `Free`
  - `TravelToQueue`: agent has cargo, traveling to back of delivery queue line
  - `Queuing`: agent physically in queue slot, shuffling forward each tick
  - `QueueManager` transitions: Loading → TravelToQueue (policy picks queue), TravelToQueue → Queuing (arrival at queue cell), Queuing → TravelLoaded (promotion when slot[0] + delivery free)
- `TaskScheduler` trait: `fn assign_pickup(zones, pos, occupied, rng)` + `fn assign_delivery(...)`
- `ActiveScheduler` resource: `Box<dyn TaskScheduler>` — 4 schedulers: random, closest, balanced, roundtrip
- `LifelongConfig` resource: tasks_completed, task_limit, throughput window
- Systems: `recycle_goals` (state machine) → `QueueManager::tick()` → solver replan
- `LogicalAgent.task_leg`: tracks current leg per agent
- **Visual mode** (`TaskStateMode` in `GraphicsConfig`):
  - **Simple** (default): 4 macro colors — Idle (grey), Picking (amber), Delivering (teal), Faulted (red/purple)
  - **Detailed**: full 8-state palette with unique color per TaskLeg
  - Toggle: Settings → "Detailed task states" checkbox
  - Internal `TaskLeg` enum always has 8 variants — visual mode only affects rendering

## Rendering Conventions

- **MaterialPalette** (`src/render/animator.rs`): pre-allocated shared handles. Color updates = handle swap, not asset mutation. Enables GPU batching. 8 task-state colors: free (grey), travel_empty (amber), loading (burnt orange), travel_to_queue (teal), queuing (deep teal), travel_loaded (green), unloading (lime), charging (blue). Special: latency (purple), dead (dark red), selected (white). Priority: selected > dead > latency > heat > task_state. 4 palettes: {Simple, Detailed} × {Normal, Colorblind} — selected by `apply_visual_palette` system in `graphics.rs`.
- **AgentRegistry** (`src/core/agent.rs`): `HashMap`-based for O(1) entity↔index lookups.
- **Orbit camera** (`src/render/orbit_camera.rs`): spherical coords (yaw/pitch/distance). Bridge: `set_camera_preset "top"|"side"`.
- **Picking** (`src/render/picking.rs`): click-to-select (ClickSelection resource) + hover highlight (HoverHighlight resource + HoverTile entity). Raycasts cursor → Y=0 plane → grid cell. Context menu in HTML.
- **Heatmap**: dual mode — density (decaying warm gradient) + traffic (cumulative blue). `DENSITY_MIN_THRESHOLD=1.8` filters single-robot cells.
- **Shadows disabled** (`shadows_enabled: false`) — eliminates extra render pass.

## Structural Rules

- **WASM compat**: No heavy multithreading, no unbuffered file I/O. Use `#[cfg(not(target_arch = "wasm32"))]` guards when needed.
- **Determinism**: All random events via `SeededRng`. Same seed + config = identical simulation.
- **Performance**: Zero-cost abstractions, cache locality, minimal Bevy change detection — only call `get_mut()` on resources that actually changed (`active_tiles` pattern).
- **ECS discipline**: Pathfinding, Fault, Rendering stay in their `SystemSet`. Execution order is source of truth.
- **No magic numbers**: All limits in `constants.rs`.

## Key Design Decisions

| Decision | Rationale |
|----------|-----------|
| Binary WASM target (not cdylib) | cdylib event loop never starts |
| HTML/CSS/JS UI (not Egui) | Full web integration, CSS theming, embeddable in Astro |
| uPlot for charts | 35 KB, canvas-native, CSS-themeable |
| Instanced rendering | 500 agents in 1 draw call |
| `SeededRng` only | Deterministic replay for research |
| Lifelong-only (not one-shot) | Fault resilience requires sustained operation |
| 5 lifelong solvers (PIBT + 3 RHCR + Token Passing) | Only lifelong-capable solvers; one-shot solvers archived |
| Scheduler as research variable | Operators choose schedulers; more actionable than solver comparison |
